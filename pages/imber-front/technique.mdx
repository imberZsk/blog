# 技术选型？

- 做项目的第一步是思考项目技术选型
- 一般公司或者部门有自己的技术栈，可以根据公司以往沉淀来做选型
- 如果要尝试使用不一样的技术，我们可以做一个 `ppt` 来给上面演示这样做的好处

## Vue和React

- react写得不规范容易报错卡死，react风格较多不同人写的代码可能天差地别，react容易出现性能问题比如重复渲染等情况，vue不用太关注性能问题，可以把组件分细一点
- 因为hooks基于单链网页容易挂，缺少依赖等情况容易出现无限循环

## Nextjs

SSR框架有这几类比较出名，基于Angular的Angular Universal(3.9k star)，基于Vue的Nuxt.js(41.8k star)，和基于React的Nextjs(98.7k star)，相信如果使用SSR框架的话，Nextjs一定是首选



其实 Nextjs 称为 SSR 不太准确，应该叫做`Ismorphic render同构渲染`，是服务端和客户端共同渲染生成页面，而不是服务端生成完整的 HTMl 页面



资源

- [贴心官网](https://nextjs.org/)
- [官方案例](https://github.com/vercel/next.js/tree/canary/examples)

功能

- 开箱即用sass/css module，基于文件的路由
- 内置标签：Head，Image，Scipt等
- 数据传递函数：getStaticProps和getStaticPaths，getServerSideProps和getInitialProps
- 钩子：useRouter
- API路由

## Nextjs 相对于 React 项目有什么好处？

SSR 不是一个新技术，早些时候是用 php/jsp 做服务端渲染，然后变成 vue/react 客户端渲染，那为什么又回到服务端渲染了呢？

- [CLS](https://web.dev/cls/)累积布局偏移：`Nextjs` 会使图片始终包含尺寸属性，利于 `CLS`
- SSR 服务端渲染：在服务端生成 `HTMl` 标签，再通过同构技术加载数据，可以加快首屏渲染，利于 `SEO`，如果不太理解为什么更快，可以看这篇文章： [为什么 SSR 比 CSR 首屏渲染更快？](https://baidu.com)
- SSG 静态站点生成：当每个用户看到的都是一样的静态页面，`SSG` 只在打包的时候构建一次生成一个 `HTML`
- 相比于老技术模版渲染，`Nextjs` 能给我们 `SPA` 一样的开发体验，还能使用客户端路由功能
- ...

## Tailwindcss/Windicss/Unocss 如何选择？

开源一个项目相对简单，长期维护一个项目却没那么容易（此时的时间是笔者在2023/01/04看到的数据），从这些数据可以看到，`Tailwindcss`将是一个不错的选择，而且使用`Nextjs`官网教授兼初始化一个项目，默认也是`Tailwindcss`

- 从github star看：Tailwindcss（63.6k）> UnoCSS（8.2k）> Windicss（6k）
- 从vscode插件下载量来看：Tailwindcss（2,286,465）>WindiCSS（75814）> UnoCSS（30323）

## Turbopack/Webpack/vite 如何选择?

前端工具链用`rust`重写已经是不争的事实，`Turbopack，Turborepo，SWR，SWC...`
对于`Webapck`，如果是公司项目，那比较推荐，对于``Vite`，`Nextjs`官方暂未配合，所以我们也最好不用

对于打包工具，可以理解为一个黑盒，我们不用去关心它内部的原理，内部是如何运作的，只需要知道和使用它的配置，给我们带来打包后需要的产物，对于学习来说，我们完全可以使用`Turbopack`尝鲜，注意不要用于生产环境，官网也有说明暂未准备好生产环境

## jsx 和 tsx 如何选择？

问题可以变为`ts`的好处

- `ts`强类型能帮助即时检测到**bug**的产生，在编辑器里就能报错，而不用等到浏览器控制台才看到报错
- `ts`利于大型项目后期维护，后期维护的时候，修改东西会有类型提示等，是利于维护的
- ...
